## はじめに

松の事例は国も教えてくれるが、竹や梅は決して教えてくれない。

有価証券報告書の開示を担当したことのある方なら、この非対称性に覚えがあるはずです。金融庁の開示事例集には模範的な100点事例が並んでいます。しかし、ちょうど法令要件を満たす60点の書き方、業界標準水準の80点の表現が「具体的にどういうものか」は、誰も教えてくれない。コンサルタントに頼めば費用が発生し、担当者が自力で調べれば膨大な工数がかかります。

この格差を解消するためのOSSを公開しました。

**disclosure-multiagent** — 有報PDFをドロップするだけで、最新法令との開示ギャップを自動検出し、梅（60点）・竹（80点）・松（100点）の3段階で改善文案を生成するAIパイプラインです。

GitHubリポジトリ: https://github.com/Majiro-ns/disclosure-multiagent

---

## 1. 問題の構造

有報の開示対応が難しい理由は、情報の非対称にあります。

法令改正は毎年行われます。2024〜2026年だけを見ても、人的資本開示の拡充、給与開示の義務化、SSBJ基準への対応、TCFD整合の開示強化と、重要な改正が連続しました。それぞれの改正ごとに「何が変わったか」「自社の有報にどう影響するか」を担当者が把握し続けることは、構造的に難しくなっています。

問題の核心は網羅性です。有報は数百ページに及ぶ文書であり、数十から百件規模の法令要件と照合する作業は、熟練した担当者でも漏れが発生します。さらに、「従業員エンゲージメント指標の開示が必要」という要件に対して、具体的に何を何文字書けば法令要件を満たすのかを、法令文そのものから判断するのは容易ではありません。

この問題は、AIと自動化が最も直接的に効果を発揮できる領域です。

## 2. disclosure-multiagentが解決すること

このツールは5段階のパイプラインで動作します。

```
有報PDF
  ↓
PDF読取   — 有報のセクションを自動抽出
  ↓
法令確認  — 最新の開示規制YAMLと照合
  ↓
ギャップ分析 — 何が足りないかを検出
  ↓
改善提案  — 梅・竹・松 3段階の記載文案を生成
  ↓
レポート生成 — Markdownレポートとして出力
```

各ステップはPythonモジュールとして独立しており、ライブラリとしても、CLIツールとしても、WebUIとしても使えます。

**最も重要な設計上の判断**は、法令情報をYAMLで管理している点です。法令改正があってもコードを変更する必要がなく、YAMLに改正内容を追記するだけで対応できます。これにより、「今年の改正に対応したか」の管理が明示的になります。

ギャップ分析の出力には根拠IDが付与されます。「AIがそう言っているから」ではなく、「この法令のこの条文が根拠」と示すことで、実務担当者が裏付けを確認できます。LLMの幻覚を抑制するために、キーワードマッチによる一次フィルタ、文脈判定、根拠ID付与、免責事項の必須出力という4層の構造を採用しています。

## 3. APIキー不要で今すぐ試す

このツールはモックモードを内蔵しており、AnthropicのAPIキーなしで動作確認ができます。

**Pythonライブラリとして**

```bash
pip install disclosure-multiagent
```

```python
import os
os.environ.setdefault("USE_MOCK_LLM", "true")

from scripts.m1_pdf_agent import extract_report
report = extract_report("your_report.pdf")
print(report.company_name, "—", len(report.sections), "sections extracted")
```

**CLIとして**

```bash
disclosure-check your_report.pdf --level 竹
```

**WebUI（Docker）として**

```bash
git clone https://github.com/Majiro-ns/disclosure-multiagent.git
cd disclosure-multiagent
cp .env.example .env
docker compose up --build
```

起動後、ブラウザで http://localhost:3010 を開くとPDFのドラッグ&ドロップ画面が表示されます。http://localhost:3010/sample では、架空企業のサンプルレポートをAPIキーなし・PDF不要で即時確認できます。

## 4. 出力例

竹（業界標準）レベルで分析した場合、以下のような改善提案が生成されます。

**検出されたギャップ: 従業員エンゲージメント指標の開示**

- 根拠: 企業内容等の開示に関する内閣府令（2024年改正）
- 改善提案（竹）:

> 当社は、従業員エンゲージメントを経営の重要指標と位置づけ、年1回の全社サーベイを実施しております。直近のエンゲージメントスコアは[数値]点であり、前年度比[増減]点の[改善/低下]となりました。

プレースホルダが残っている箇所には警告マークが付与され、担当者が埋め忘れるリスクを低減します。

法令情報が更新されるたびに、このような改善提案が自動的に生成されます。担当者が毎年行う法令改正の追跡作業を、YAMLの更新という明示的な管理に置き換えることが、このツールの本来の役割です。

## おわりに

このOSSが目指すのは、60点でハードルを確実に超える開示を、コンサルタントなしで展開できる状態を作ることです。

有報開示の現場では、100点を目指すリソースがない企業が、60点と100点の間で迷い続けることがあります。法令要件を確実にクリアできる水準とはどこかが不明なまま、過剰対応と対応漏れの間を手探りする状況です。

梅・竹・松という3段階の提案を出力するのは、この迷いを構造的に解消するためです。「どこまでやれば十分か」に対して、根拠付きで答えを出せるツールを作ることが、開発の出発点でした。
