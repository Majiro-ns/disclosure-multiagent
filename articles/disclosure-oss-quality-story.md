# テスト735件PASSのOSSが実データで3点だった

> [GitHubリポジトリ](https://github.com/Majiro-ns/disclosure-multiagent) | [関連記事: 開発実録](https://zenn.dev/majiro_ns/articles/disclosure-oss-intro)

## はじめに

735件のテストが通っている状態で、実データを入れたらスコア3.0/10でした。

筆者はAIマルチエージェントシステムで複数の経理OSSを並行開発しています。有価証券報告書の開示品質を自動検査する [disclosure-multiagent](https://github.com/Majiro-ns/disclosure-multiagent) もその一つです。金融庁ガイドラインとの照合、人的資本開示の充足度評価、法令ベースの改善提案生成。一通り実装し、CIは緑、E2Eも通過していました。

**公開直前に実データを入れたら、全てが崩れました。**

この記事で得られることは3つあります。

- テストが通ることと正しく動くことの間にある溝の実例
- 実データ投入で発覚した4つの設計欠陥とその原因箇所
- それでもOSSを公開した判断の根拠

30分で確信が崩れた記録です。

### この記事の対象読者

- LLMを組み込んだアプリケーションを開発している方
- テストは通るのに本番で壊れた経験がある方
- 有価証券報告書・決算短信を扱うシステムに興味がある方

## 1. パイプラインの構造

disclosure-multiagentは5段階のパイプラインで動きます。

```text
M1（PDF解析）→ M2（法令マッピング）→ M3（ギャップ分析）→ M4（提案生成）→ M5（レポート出力）
```

有報PDFを入力とし、法令YAMLファイル群と照合して開示上の不足を検出します。出力は梅・竹・松の3段階チェックレポートです。

> 「**松の事例は国も教えてくれるが、竹や梅は決して教えてくれない。**」

| 水準 | スコア | 説明 |
|------|--------|------|
| 梅 | 60点 | **法令準拠ライン** — これだけやれば監督官庁から指摘されない |
| 竹 | 80点 | **業界標準** — 同業他社と遜色ない水準 |
| 松 | 100点 | **先進開示** — 機関投資家から評価される水準 |

このうち梅が基本、松が詳細分析です。

テストスイートは735件。単体・統合・E2Eが揃っており、CIも通っていました。

この構造の中に、テストでは見えない穴が4つ埋まっていました。

## 2. 耐火物メーカーのPDFを入れた

実験対象に選んだのは、耐火物メーカーの第3四半期決算短信です。製鋼用耐火物とファインセラミックスの製造会社。J-GAAP適用。手元にあったから選びました。有報でも決算短信でも処理できるはず、という思い込みがありました。

パイプラインを実行します。エラーは出ません。レポートが生成されました。

```yaml
pipeline_execution:
  ume:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
  take:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
  matsu:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
```

32セクション抽出されています。しかしギャップ分析を通過したのは1セクションだけです。そして梅・竹・松で total_gaps が完全に一致している。レベル別に変わるはずの提案件数が、同じ1323を返しています。

ここから先が、30分間の話です。

---

## 3. 定型文が1301件

1323件の改善提案を開くと、全セクション名が (2) 連結財政状態 でした。

原因は `m3_gap_analysis_agent.py` の `is_relevant_section()` にあります。

```python
def is_relevant_section(section: SectionData) -> bool:
    all_keywords = (
        HUMAN_CAPITAL_KEYWORDS   # 人的資本関連 30語
        + SSBJ_KEYWORDS          # サステナビリティ 15語
        + BANKING_KEYWORDS       # 銀行業 10語
    )
    return any(kw in section.heading for kw in all_keywords)
```

32セクション中31を除外していました。耐火物メーカーの見出しには人的資本やサステナビリティのキーワードがほぼ登場しません。唯一、BANKING_KEYWORDSの信用リスクが財政状態の見出しに部分一致して、それだけが通過しました。

残った1セクションに対して全法令エントリ × 全開示チェック項目が掛け合わされ、1323件が生成されました。内訳を見ると1301件が同一の定型文です。

> 当社は、法令の定めに従い人材・環境に関する事項を適切に対応・開示しています。

会社名も業種も数値も入っていません。耐火物を作っている会社の固有性がゼロの文が1301件、レポートを埋めていました。定型文率98.3%です。

ホワイトリスト方式の設計は、想定した業種には機能します。しかし想定外の業種が来た瞬間に、ほぼ全てを弾く。配管が1本しかない建物と同じで、その1本が詰まれば全体が止まります。

## 4. 耐火物メーカーにバーゼルIII規制

定型文の群れを読み飛ばしていくと、異質な提案が混じっていました。

> CET1比率は[CET1比率]%、バーゼルIII規制水準を充足しています。（参照: ias_1_2025.yaml）

CET1比率は銀行の自己資本比率の指標です。バーゼルIIIは国際決済銀行が定める銀行規制フレームワークです。これが耐火物メーカーへの改善提案として出力されていました。

原因は `laws/ias_1_2025.yaml` にあります。

```yaml
- id: "ias_1_capital_management"
  category: "IFRS（財務諸表表示）"
  disclosure_items:
    - "CET1比率（バーゼルIII規制水準の充足状況）"
  applicable_industries: []  # 全業種対象になっている
```

`applicable_industries: []` は全業種適用を意味します。銀行規制の指標が業種を問わず適用される設定でした。

さらに `load_law_context()` に `accounting_standard` パラメータがありません。J-GAAP企業にもIFRS法令YAMLが全件読み込まれます。血液型を確認せずに全種類の輸血パックを接続するような状態です。レポートにはIFRS関連のURL警告が8件出力されており、ツール自身が信頼性未確認だと申告していました。

## 5. 何を渡しても有報として処理される

最も根本的な欠陥は、入力PDFの種別を判定する機能がなかったことです。

入力したのは第3四半期決算短信です。1ページ目にそう書いてあります。しかしパイプラインはこれを有価証券報告書として処理していました。

```python
def extract_report(pdf_path: str, doc_type: str = "yuho") -> StructuredReport:
    # doc_typeのデフォルトが "yuho"（有報）固定
    ...
```

`doc_type` を渡さなければ、何を入力しても有報として処理します。有報は年次の詳細開示書類で人的資本やサステナビリティの記載義務があります。決算短信は四半期の業績速報で、構造も記載要件も全く異なります。

有報専用ツールが、入力の書類種別を確認しない。症状を聞かずに処方箋を出す構造です。その結果、決算短信に存在しない人的資本開示の欠落を1323件指摘することになりました。

## 7. それでも公開した

結論から書くと、条件付きで公開しました。

今回の実行は `USE_MOCK_LLM=true` で動いています。実際のLLMを呼んでいません。モック出力は固定値なので、梅・竹・松でギャップ数が一致するのは仕様上の帰結です。実LLMであれば提案内容は変わります。

4つのCRITバグ全てについて、原因箇所と修正設計が特定されています。

- CRIT-01（書類種別判定）: `detect_doc_type()` 関数の追加
- CRIT-02（セクションフィルタ）: ホワイトリストからブラックリスト方式への転換
- CRIT-03（銀行規制混入）: YAMLの適用業種タグ追加
- CRIT-04（会計基準フィルタ）: `accounting_standard` パラメータ追加

バグの所在が分かっていて、修正設計もあって、スコアリングの枠組み自体は動いている。であれば、閉じたまま磨き続けるより公開して直す方が速い、という判断です。

2026-03-28追記: CRIT-01とCRIT-02を修正済みです。新規テスト12件追加、テスト総数735→747件。CRIT-03・CRIT-04は継続対応中です。

## おわりに

テストが通ることと正しく動くことの間には、実データの複雑さという溝があります。

735件のテストはパイプラインの構造を検証しています。各ステージが入力を受け取り、期待された形式で出力することは確認できます。しかしモックで組んだテストは、耐火物メーカーのPDFを入力したとき何が起きるかを教えてくれません。

今回残った問いは3つあります。

- 書類種別を自動判定すべきか、ユーザーに指定させるべきか
- 業種フィルタをメタデータで管理するか、ディレクトリ構造で管理するか
- モックと実LLMの乖離をどうテストで担保するか

これらはテストを書く前に答えるべき設計の問いでした。実データに通して初めて、問い自体の存在に気づいた。その記録です。

**もしあなたが同じ溝に直面しているなら、issueやPRを歓迎します。**

### 試してみる

APIキー不要で今すぐ動かせます。

```bash
git clone https://github.com/Majiro-ns/disclosure-multiagent.git
cd disclosure-multiagent
pip install -e ".[dev]"
disclosure-check your_report.pdf --level 竹
```

`USE_MOCK_LLM=true`（デフォルト）でモックLLMが起動し、パイプライン全体が動作します。
