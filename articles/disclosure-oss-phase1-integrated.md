## はじめに

有価証券報告書（以下「有報」）の作成は、上場企業にとって毎年度の重大義務です。しかし、2024〜2026年にかけて金融庁が矢継ぎ早に打ち出した人的資本・サステナビリティ開示の法令改正により、「どの項目が追加になったか」「自社の記載が対応済みか」を人手で確認する作業のコストが急激に増大しています。

本記事では、この課題に対して**Pythonマルチエージェントシステム（M1〜M5パイプライン）**を構築し、トヨタ自動車の実際の有報に対して**7件の開示漏れ・改善点を自動検出**した実装と結果を紹介します。

---

## 1. 課題：有報の法令対応コストと人手レビューの限界

### 1.1 法令改正の連続と対応コストの肥大化

有報の開示要件は近年、急速に変化しています。金融庁は2025〜2026年にかけて以下の主要改正を実施しました。

- **企業内容等の開示に関する内閣府令改正（2026年2月）**: 人的資本開示拡充・給与開示の義務化。「従業員給与等の決定に関する方針」「平均年間給与の対前事業年度増減率（連結・単体両方）」が新たに必須記載事項に追加。
- **金融庁「人的資本の開示等に関するWG」好事例集（2025年4月）**: 人材戦略と経営戦略の連動、定量的KPI・目標値の設定、ガバナンス体制の開示などが「好事例」として示され、実質的なスタンダードを形成。
- **人的資本可視化指針の見直し（内閣官房・金融庁・経済産業省）**: サステナビリティ関連リスク・機会の開示強化。

これらの改正が毎年重なると、「昨年と何が変わったか」を把握するだけで担当者は相当な工数を要します。大企業であれば法務・IR部門が対応しますが、中堅・中小上場企業では専任担当者が限られており、見落としリスクが高まっています。

### 1.2 人手レビューの3つの限界

現状の人手レビューには構造的な問題があります。

**① 網羅性の欠如**: 有報は数百ページに及ぶため、全セクションを法令改正と照合する作業は現実的ではありません。担当者の経験や記憶に依存した「抜け」が発生します。

**② 更新頻度への追随困難**: 法令改正は年複数回行われます。前回改正の対応が完了しないうちに次の改正が来るサイクルでは、人手による追随に限界があります。

**③ 定量基準の曖昧さ**: 「人材戦略と経営戦略の連動を明示すること」のような要件は、何をどう書けば「対応済み」かが明確でないケースがあります。過去の記載例や好事例との比較が必要ですが、それを毎回人手で行うのは非効率です。

### 1.3 解決アプローチ：マルチエージェントによる自動検出

これらの課題に対し、我々は以下のアプローチを採りました。

1. **法令情報のYAML化**: 各法令改正・ガイダンスの要求事項を検出キーワードと共にYAMLで管理
2. **M1〜M5のパイプライン**: PDF解析→法令照合→ギャップ分析→提案文生成→レポート出力
3. **LLMの活用と幻覚対策**: Claude Haiku を活用しつつ、4層の幻覚防止機構を実装
4. **テスト駆動**: 747テスト全件PASSで品質を担保

本システムは tmux + Claude Code を活用したマルチエージェント並列開発フレームワークを用いて実装しました。複数の AI エージェントが M1〜M5 の各モジュールを並列で開発・検証しています。

「テストが通る ≠ 正しい」の原則のもと、三段階の品質ゲート（実装エージェントのセルフレビュー → レビューエージェントのスポットチェック → 最終品質検査）を適用しています。マルチエージェントで開発されたシステムを、マルチエージェントが品質管理するという構造です。

## 3. 実装詳細：各エージェントの設計と工夫

### 3.1 M1: PDF解析エージェント

有報PDFの解析には**PyMuPDF（fitz）**を使用しています。

> ※ 以下は設計概念を示す擬似コードです。実際のメインAPIは `extract_report()` 関数（`scripts/m1_pdf_agent.py`）として実装されており、`extract_sections` という名前の関数は存在しません。内部ヘルパー関数（`generate_doc_id` / `extract_company_name` / `extract_fiscal_year`）はプライベート実装（`_make_document_id()` / `_extract_company_name()`）として実装されています。

```python
import fitz  # PyMuPDF

def extract_sections(pdf_path: str) -> StructuredReport:
    doc = fitz.open(pdf_path)
    full_text = "\n".join(page.get_text() for page in doc)
    sections = split_sections_from_text(full_text)
    return StructuredReport(
        document_id=generate_doc_id(pdf_path),
        company_name=extract_company_name(full_text),
        fiscal_year=extract_fiscal_year(full_text),
        sections=sections,
    )
```

**設計上の工夫**: M1はPDF解析エラーが発生しても例外を raise せず、`WARNING` を出力して空sectionsの `StructuredReport` を返す**graceful degradation**設計にしています。これにより、不完全なPDFでもパイプライン全体が止まりません。

```
WARNING m1_pdf_agent:m1_pdf_agent.py:263 PDF開封エラー（空sectionsで継続）: '.' is no file
```

**パフォーマンス**: 10社の有報PDFで検証した結果、平均処理時間は**0.828秒/社**でした。ファンド型を自動除外するロジックも実装しており、投資信託の運用報告書など有報以外のPDFを誤処理しない設計になっています。

**セクション分割**: 13のキーワード（「第一部」「企業情報」「第二部」「財務情報」「人的資本」等）を正規表現でマッチングし、有報の標準構成に沿ったセクション分割を行います。

### 3.2 M2: 法令照合エージェント

法令情報はYAMLファイルで管理しています。

```yaml
# laws/human_capital_2024.yaml（実際のスキーマ）
amendments:
  - id: "hc-2026-001"
    category: "人的資本"
    change_type: "追加必須"
    effective_from: "2026-02-20"
    title: "企業内容等の開示に関する内閣府令改正（人的資本開示拡充・給与開示）"
    source: "https://sustainablejapan.jp/2026/02/23/fsa-ssbj-4/122214"
    required_items:
      - "企業戦略と関連付けた人材戦略の記載"
      - "従業員給与等の決定に関する方針の記載"
      - "平均年間給与の対前事業年度増減率の記載（連結・単体）"
```

YAMLを正データとする設計により、法令改正があっても**コードを変更せずYAMLを追記するだけ**で対応できます。M6（e-Gov API連携）でYAMLのURLを自動補完する仕組みも別途実装しています。

### 3.3 M3: ギャップ分析エージェント（幻覚対策4層構造）

M3はパイプラインの核心部分であり、最も幻覚が発生しやすいモジュールです。以下の4層構造で精度を担保しています。

**第1層: キーワードマッチによる一次フィルタ**

> ※ 以下は設計概念を示す擬似コードです。実際のキーワード判定は `judge_gap()` 関数（`scripts/m3_gap_analysis_agent.py`）の内部処理として実装されており、`_keyword_check` という名前の独立した公開関数は存在しません。

```python
# 【擬似コード: 設計概念の説明用】
def _keyword_check(section_text: str, keyword: str) -> bool:
    """キーワードが本文に存在するか確認（大文字小文字・全半角を正規化）"""
    normalized_text = unicodedata.normalize("NFKC", section_text)
    normalized_kw = unicodedata.normalize("NFKC", keyword)
    return normalized_kw in normalized_text
```

キーワードが存在しない場合は「未記載の可能性あり」として次層に渡します。キーワードが存在する場合はギャップなしと判定し、LLMを呼び出しません（**コスト最適化**）。

**第2層: LLMによる文脈判定（USE_MOCK_LLMで切替可能）**

キーワードマッチでは「あるが不十分」なケースを検出できません。第2層ではClaude Haikuが文脈を読んで「要件を満たしているか」を判定します。

> ※ 以下は設計概念を示す擬似コードです。実際のLLM判定は `judge_gap()` 関数（`scripts/m3_gap_analysis_agent.py` L325）に統合されており、`analyze_gap_with_llm` という独立した関数は存在しません。

```python
# 【擬似コード: 設計概念の説明用】
# 実装シグネチャ: judge_gap(section, disclosure_item, law_entry, client, use_mock=False)
use_mock = os.environ.get("USE_MOCK_LLM", "").lower() in ("true", "1", "yes")
# use_mock=True のときはモック応答（APIキー不要）
# use_mock=False（デフォルト: 環境変数未設定）のときは実Claude Haiku API呼び出し
```

`USE_MOCK_LLM`環境変数は**デフォルト未設定（本番モード）**です。テスト時は`USE_MOCK_LLM=true`を明示設定することでAPIキーなしで全テストが通ります（747 passed）。本番環境では`USE_MOCK_LLM`を設定しないことで実LLM呼び出しが行われます。

**第3層: 根拠ID付き出力**

全てのギャップ指摘に`根拠ID`（`HC_20260220_001`等）を付与し、どのYAMLエントリが根拠かを明示します。「AIが言っているから」ではなく「この法令のこの条文が根拠」と示すことで、実務担当者が裏付けを確認できます。

**第4層: 信頼度スコアと免責事項**

レポート冒頭に免責事項を必須出力し、「本レポートは参考情報です。最終判断は専門家にご依頼ください」と明示しています。

### 3.4 M4: 松竹梅提案エージェント

ギャップが検出された項目に対して、3段階のレベルで改善提案文を生成します。

| レベル | 対象 | 特徴 |
|--------|------|------|
| 松 | 先進的開示を目指す大企業 | 数値目標・KPI・国際基準（ISSB/GRI）準拠の詳細記載 |
| 竹 | スタンダードな実務対応 | 必須要件を満たす標準的な記載。プレースホルダ付き |
| 梅 | 最低限の法令対応 | コンパクトな記載で法令要件のみクリア |

```python
FEW_SHOT_EXAMPLES = {
    "従業員給与等の決定に関する方針": {
        "松": (
            "当社の給与水準は、職務の難易度・業績貢献・市場相場の三要素を基本に決定しています。\n\n"
            "[方針]\n...（詳細な方針・KPI・国際基準準拠の記載）"
        ),
        "竹": (
            "当社の従業員給与は、職務の難易度・スキル・業績への貢献度、"
            "および市場相場を踏まえて決定しています。賞与は会社業績と個人評価を"
            "反映した連動制を採用しています。"
        ),
        "梅": "当社の従業員給与は、職務・スキル・業績を踏まえ決定しています。",
    },
    "GHG排出量（Scope1・Scope2）の開示": {
        "松": "...",
        "竹": "...",
        "梅": "...",
    },
    # （以下、各セクション名に対する松竹梅の例文が続く）
}

def generate_proposal(
    section_name: str,
    change_type: str,
    law_summary: str,
    law_id: str,
    level: str,
    system_prompt: Optional[str] = None,
) -> str:
    prompt = _build_few_shot_prompt(section_name, change_type, law_summary, level, FEW_SHOT_EXAMPLES)
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
```

few-shotプロンプトにより、出力が実際の有報文体に近くなります。プレースホルダ（`[平均年間給与額]`等）が残っている場合は⚠️マークで明示し、担当者が埋め忘れるリスクを低減します。

### 3.5 M5: レポート統合エージェント

M1〜M4の出力を統合し、Markdown形式のレポートを生成します。

**出力形式**:
1. 免責事項（必須）
2. 分析メタ情報（対象期間・提案レベル・法令情報取得日）
3. 変更箇所サマリ（変更種別×件数の表）
4. セクション別の変更提案（変更種別・対象項目・根拠・提案文）
5. 参照した法令・ガイダンス一覧

さらに、**Streamlit UI**も実装しており、PDFをアップロードするとブラウザ上でレポートを確認できます。

Streamlit環境とCLI環境の両対応は `scripts/app.py` の以下のガードで実現しています。

```python
# scripts/app.py
def _is_streamlit_running() -> bool:
    """Streamlit のアクティブセッション内で実行されているか確認する。"""
    try:
        from streamlit.runtime.scriptrunner import get_script_run_ctx
        return get_script_run_ctx() is not None
    except Exception:
        return False

if _is_streamlit_running():
    main()
```

`streamlit run app.py` での実行時は `get_script_run_ctx()` が非Noneを返して `main()` が呼ばれ、`python3 -c "import app"` でのインポート時はNoneを返してスキップされます。これによりCLI実行とWebUI実行の両方に対応しています。

### 4.5 結果の解釈

今回の7件は、主に**2026年2月の法令改正（3件）**と**2025年4月の好事例集（4件）**に由来します。システムは「現行の有報テキストに指定キーワードが存在するか」を一次判定しているため、偽陽性（実際には対応済みなのに指摘）が発生する可能性があります。実務では、本レポートを出発点として人間が精査するフローを推奨します。

## 5. 実データで3点だった — テスト735件PASSの裏側

### 5.1 耐火物メーカーのPDFを入れた

トヨタの有報では7件を検出できました。では他の企業ではどうか。

実験対象に選んだのは、耐火物メーカーの第3四半期決算短信です。製鋼用耐火物とファインセラミックスの製造会社。J-GAAP適用。手元にあったから選びました。

パイプラインを実行します。エラーは出ません。レポートが生成されました。

```yaml
pipeline_execution:
  ume:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
  take:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
  matsu:
    m1_sections_extracted: 32
    m3_sections_analyzed: 1
    total_gaps: 1323
```

32セクション抽出されています。しかしギャップ分析を通過したのは1セクションだけです。そして梅・竹・松で total_gaps が完全に一致している。レベル別に変わるはずの提案件数が、同じ1323を返していました。

### 5.2 定型文が1301件（98.3%）

1323件の改善提案を開くと、全セクション名が `(2) 連結財政状態` でした。

原因は M3 の `is_relevant_section()` にあります。32セクション中31を除外していました。耐火物メーカーの見出しには人的資本やサステナビリティのキーワードがほぼ登場しません。唯一、BANKING_KEYWORDSの「信用リスク」が財政状態の見出しに部分一致して、それだけが通過しました。

残った1セクションに対して全法令エントリ × 全開示チェック項目が掛け合わされ、1323件が生成されました。内訳を見ると1301件が同一の定型文です。

> 当社は、法令の定めに従い人材・環境に関する事項を適切に対応・開示しています。

会社名も業種も数値も入っていない。耐火物を作っている会社の固有性がゼロの文が1301件、レポートを埋めていました。

### 5.3 耐火物メーカーにバーゼルIII規制

定型文の群れを読み飛ばしていくと、異質な提案が混じっていました。

> CET1比率は[CET1比率]%、バーゼルIII規制水準を充足しています。

CET1比率は銀行の自己資本比率の指標です。バーゼルIIIは国際決済銀行が定める銀行規制フレームワークです。これが耐火物メーカーへの改善提案として出力されていました。

原因は法令YAMLの `applicable_industries: []`（全業種適用）と、`load_law_context()` に会計基準フィルタがないこと。J-GAAP企業にもIFRS法令YAMLが全件読み込まれる状態でした。

### 5.4 何を渡しても有報として処理される

最も根本的な欠陥は、入力PDFの種別を判定する機能がなかったことです。

```python
def extract_report(pdf_path: str, doc_type: str = "yuho") -> StructuredReport:
    # doc_typeのデフォルトが "yuho"（有報）固定
```

入力したのは第3四半期決算短信です。1ページ目にそう書いてあります。しかしパイプラインはこれを有価証券報告書として処理していました。有報専用ツールが、入力の書類種別を確認しない。その結果、決算短信に存在しない人的資本開示の欠落を1323件指摘することになりました。

## 6. それでも公開した — 4つのCRITバグと修正

### 6.1 修正設計

4つのCRITバグ全てについて、原因箇所と修正設計を特定しました。

- **CRIT-01（書類種別判定）**: `detect_doc_type()` 関数の追加
- **CRIT-02（セクションフィルタ）**: ホワイトリストからブラックリスト方式への転換
- **CRIT-03（銀行規制混入）**: YAMLの適用業種タグ追加
- **CRIT-04（会計基準フィルタ）**: `accounting_standard` パラメータ追加

バグの所在が分かっていて、修正設計もあって、スコアリングの枠組み自体は動いている。であれば、閉じたまま磨き続けるより公開して直す方が速い、という判断です。

CRIT-01とCRIT-02は修正済みです。新規テスト12件追加、テスト総数735→747件。CRIT-03・CRIT-04は継続対応中です。

## おわりに

本記事では、有価証券報告書の開示漏れをAIで自動検出するPythonマルチエージェントシステム（disclosure-multiagent）の設計・実装・デモ結果を紹介しました。

トヨタ自動車の有報で7件の開示漏れを検出し、M1〜M5のパイプラインが動くところまでは順調でした。しかし耐火物メーカーの決算短信を入れた瞬間、定型文率98.3%、バーゼルIII規制の混入、書類種別判定の欠如 — テストでは見えなかった4つの設計欠陥が露呈し、スコア3.0/10を記録しました。

テストが通ることと正しく動くことの間には、実データの複雑さという溝があります。735件のテストはパイプラインの構造を検証しています。しかしモックで組んだテストは、想定外の業種のPDFを入力したとき何が起きるかを教えてくれません。

それでも公開したのは、バグの所在と修正設計が明確だったからです。CRIT-01/02は修正済み（テスト747件）、CRIT-03/04は継続対応中。閉じたまま磨き続けるより、公開して直す方が速いと判断しました。

> 「**松の事例は国も教えてくれるが、竹や梅は決して教えてくれない。**」

60点でハードルは超える開示、それを支援するOSSを目指して開発を続けています。

APIキー不要で今すぐ動かせます。

```bash
git clone https://github.com/Majiro-ns/disclosure-multiagent.git

disclosure-check your_report.pdf --level 竹
```
